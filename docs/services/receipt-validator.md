# Receipt Validator

## Overview

`IReceiptValidator` is the security gate between an IAP purchase and granting the reward. The template ships `StubReceiptValidator` which always returns `true` — fine for prototyping, **never ship it**. Real impls validate receipts server-side (your backend forwards to Apple's `verifyReceipt` or Google Play's `purchases.products.get`) or use Unity IAP's `CrossPlatformValidator` for client-side validation as a weaker fallback.

For hybrid casual where the only thing IAP grants is in-game currency consumed locally, the failure mode is "user gets free coins" — recoverable. For premium / subscription games where IAP grants persistent advantages, server-side validation is mandatory.

## Public API

```csharp
namespace Zero.Core
{
    public interface IReceiptValidator
    {
        UniTask<bool> ValidateAsync(string productId, string receipt, CancellationToken ct = default);
    }
}
```

The interface is one method on purpose — receipts come in many shapes, output is binary.

## Stub behavior

`StubReceiptValidator` (`Assets/_Project/Scripts/Runtime/Services/ReceiptValidator/StubReceiptValidator.cs`):
- `ValidateAsync(_, _)` returns `UniTask.FromResult(true)`.
- Logs `[ReceiptValidator] Stub accepting receipt for {productId}` so it's loud during dev.

## Extension Points

### Server-side validation (recommended)

The right answer for any game with revenue: your backend receives `{productId, receipt, platform}`, forwards to the appropriate verification endpoint, returns true/false.

```csharp
public sealed class ServerReceiptValidator : IReceiptValidator
{
    private readonly HttpClient _http;
    private const string Endpoint = "https://api.yourgame.example/iap/validate";

    public ServerReceiptValidator(HttpClient http) { _http = http; }

    public async UniTask<bool> ValidateAsync(string productId, string receipt, CancellationToken ct = default)
    {
        var payload = JsonConvert.SerializeObject(new
        {
            productId,
            receipt,
            platform =
#if UNITY_IOS
                "ios",
#elif UNITY_ANDROID
                "android",
#else
                "unknown",
#endif
        });
        using var content = new StringContent(payload, Encoding.UTF8, "application/json");
        try
        {
            using var res = await _http.PostAsync(Endpoint, content, ct);
            if (!res.IsSuccessStatusCode) return false;
            var body = await res.Content.ReadAsStringAsync();
            return JsonConvert.DeserializeObject<ValidationResponse>(body).Valid;
        }
        catch
        {
            // Network error — fail closed (don't grant). Log so support can retry.
            return false;
        }
    }

    private sealed class ValidationResponse { public bool Valid { get; set; } }
}
```

Replace the binding in `ReceiptValidatorServiceInstaller.cs`:
```csharp
builder.RegisterType(typeof(ServerReceiptValidator), new[] { typeof(IReceiptValidator) }, Lifetime.Singleton, Resolution.Lazy);
```

The backend implementation is out of template scope — typically a Cloud Function or Lambda calling [Apple's verifyReceipt](https://developer.apple.com/documentation/appstoreserverapi) and [Google's Purchases.Products.Get](https://developers.google.com/android-publisher/api-ref/rest/v3/purchases.products/get).

### Client-side validation via Unity IAP `CrossPlatformValidator`

Weaker (private key on the client = decompilable) but better than the stub. Unity IAP's `CrossPlatformValidator` ships `tangle` files generated by the `Window → Unity IAP → Receipt Validation Obfuscator` editor tool.

```csharp
using UnityEngine.Purchasing.Security;

public sealed class CrossPlatformReceiptValidator : IReceiptValidator
{
    private readonly CrossPlatformValidator _validator;

    public CrossPlatformReceiptValidator()
    {
        _validator = new CrossPlatformValidator(
            GooglePlayTangle.Data(),
            AppleTangle.Data(),
            Application.identifier);
    }

    public UniTask<bool> ValidateAsync(string productId, string receipt, CancellationToken ct = default)
    {
        try
        {
            _validator.Validate(receipt);
            return UniTask.FromResult(true);
        }
        catch (IAPSecurityException)
        {
            return UniTask.FromResult(false);
        }
    }
}
```

Generate the tangle files via `Window → Unity IAP → Receipt Validation Obfuscator` and check them into your private game repo (NOT this template).

### Composite validation

Belt + suspenders: client-side rejects obviously bad receipts (malformed, wrong bundle id), server-side validates against the storefront for the rest:

```csharp
public sealed class CompositeReceiptValidator : IReceiptValidator
{
    private readonly CrossPlatformReceiptValidator _client;
    private readonly ServerReceiptValidator _server;

    public async UniTask<bool> ValidateAsync(string productId, string receipt, CancellationToken ct = default)
    {
        if (!await _client.ValidateAsync(productId, receipt, ct)) return false;
        return await _server.ValidateAsync(productId, receipt, ct);
    }
}
```

## Examples

Wire into the IAP purchase flow ([iap.md](iap.md#validate-receipts)):

```csharp
public async UniTask<bool> BuyCoins100()
{
    var outcome = await _iap.PurchaseAsync("coins_100");
    if (outcome.Result != PurchaseResult.Purchased) return false;
    if (!await _validator.ValidateAsync("coins_100", outcome.Receipt)) return false;
    _wallet.Add("coins", 100);
    return true;
}
```

## Known Limitations

- **Stub accepts everything.** Logs at info level so it's visible — but don't rely on console scanning to remember to swap.
- **No retry / queue.** A failed validation is dropped on the floor. For real games, persist failed validations in `ISaveService` and retry on next launch — Apple receipts are idempotent and stay valid until refunded.
- **No refund handling.** Apple Server-to-Server notifications inform refunds; client-side validation can't see them. The server impl + a backend webhook is the right path.

## Design Rationale

- **Single-method interface** because validation is binary at the call site. Implementation complexity (server round-trip, retry, refund) is hidden.
- **`UniTask<bool>` not `UniTask<ValidationResult>`** — there's no useful detail to return at the IAP-flow boundary. Failure reasons (network, invalid receipt, refunded) are diagnostic and belong in logs / crash reports.
- **`Stub` ships rather than nothing** because the IAP flow becomes "if no validator, bypass" — that conditional is the kind of code that survives into production. A loud stub forces consumers to swap.
- **Default fail-closed in real impls** — network errors should NOT grant the reward. Hybrid casual revenue tolerates a few players retrying; it does not tolerate a refund-and-keep-coins exploit.
